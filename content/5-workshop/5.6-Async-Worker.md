## Why Asynchronous Processing

Calling an AI model to generate content takes anywhere from a few seconds to tens of seconds. If the EC2 API waited synchronously for Gemini's response before replying to the client, it would hold connections open for too long, risk timeouts, and fail to scale when many users generate content at once. The solution: the API receives the request → enqueues a job → immediately returns `202 Accepted` with a `jobId` → a separate process (the Lambda Worker) handles the AI work in the background → the client polls the job status itself.

## Amazon SQS

| Queue | Role |
| --- | --- |
| `quillo-generation-queue-prod` (Main Queue) | Receives AI jobs from the EC2 API, a Standard Queue (FIFO isn't needed since processing order doesn't matter) |
| `quillo-generation-dlq-prod` (Dead Letter Queue) | Captures messages that fail beyond the retry limit, via a redrive policy |

| Parameter | Value |
| --- | --- |
| Visibility Timeout | 300 seconds (enough time for Lambda to process one job before the message is considered "unprocessed" and redelivered) |
| Redrive Policy | `maxReceiveCount = 5` → after 5 failures, the message automatically moves to the DLQ |

## AWS Lambda Worker

The Lambda Worker is written in Node.js/TypeScript and runs **inside the VPC** (VPC-attached, with an ENI in the private app subnet) so it can reach RDS/Redis over the internal network.

| Processing step (`processMessage`) | Description |
| --- | --- |
| 1. Receive message | **Event Source Mapping** automatically polls the Main Queue and invokes Lambda when a message arrives — Lambda does not actively call SQS itself |
| 2. Check quota | Calls `checkQuota()` — if the organization has exceeded its limit, marks the job `FAILED` and returns (without throwing, to avoid pointless retries/DLQ) |
| 3. Load data | Queries `ContentPiece` + `BrandPersona` from RDS Primary via Prisma |
| 4. Build prompt | Combines the Brand Persona (tone, voice, keywords, avoid-words) with the user's brief into a complete prompt |
| 5. Call the AI provider | `ai.service.ts` dispatches based on `AI_PROVIDER=gemini` → calls Gemini 2.5 Flash/Flash-Lite |
| 6. Save the result | Transaction: creates a new `ContentVersion` (marks it `isActive=true`, sets older versions to `false`), updates the `GenerationJob` to `COMPLETED`, and writes a `UsageLog` entry (tokens consumed) |

The deployment package is built with esbuild (`build-lambda.sh`), bundling all dependencies into a single `index.js` file (bundle-all strategy) to avoid missing-module errors on the Lambda runtime.

## Gemini API Integration

- Lambda retrieves `GEMINI_API_KEY` from Secrets Manager (never hardcoded).
- Models used: **Gemini 2.5 Flash** for new generation, **Gemini 2.5 Flash-Lite** for post-processing operations (rewrite/expand/shorten) — Flash-Lite is cheaper and faster, well suited to light edits on existing content.
- Token usage is read from `usageMetadata.promptTokenCount` / `candidatesTokenCount` in the response and written to `UsageLog` to support quota tracking.
- Lambda reaches the internet (Gemini API) through the NAT Gateway (described in [5.3 Networking](../5.3-networking/)).

## Verification

```bash
aws sqs get-queue-attributes --queue-url <main-queue-url> \
  --attribute-names ApproximateNumberOfMessages ApproximateNumberOfMessagesNotVisible

aws lambda get-function --function-name quillo-worker --region ap-southeast-1 \
  --query "Configuration.[State,LastUpdateStatus]"

aws logs tail /aws/lambda/quillo-worker --since 1h --region ap-southeast-1
```

> **Expected result:** `get-queue-attributes` shows the number of pending messages dropping back to 0 once the job is processed; the Lambda command returns `State: "Active"`, `LastUpdateStatus: "Successful"`; the CloudWatch logs contain a line confirming job completion, e.g. the keyword `"status":"COMPLETED"` with no `ERROR` or `Task timed out`.
