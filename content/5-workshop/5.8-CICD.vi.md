## Xác thực AWS bằng OIDC — không dùng Access Key dài hạn

GitHub Actions xác thực với AWS thông qua **OpenID Connect (OIDC)**, loại bỏ hoàn toàn nhu cầu lưu `AWS_ACCESS_KEY_ID`/`AWS_SECRET_ACCESS_KEY` dạng GitHub Secret — đây là access key tĩnh, tồn tại vĩnh viễn cho tới khi bị revoke thủ công, rủi ro cao hơn nhiều so với credential tạm thời mà OIDC cấp cho mỗi lần chạy workflow.

| Thành phần | Cấu hình |
| --- | --- |
| OIDC Identity Provider | `token.actions.githubusercontent.com` |
| IAM Role | `quillo-github-actions-deploy-role` |
| Trust Policy | Chỉ chấp nhận token từ repo `Chubekho/Quillo`, branch `main` — workflow chạy từ branch khác hoặc fork không xin được credential |

> Lưu ý khi setup: AWS CLI **bắt buộc** tham số `--thumbprint-list` khi tạo OIDC provider, dù giá trị này không còn thực sự được GitHub Actions xác minh từ 2023 — thiếu tham số này lệnh `create-open-id-connect-provider` sẽ báo lỗi.

## IAM Policy — tách riêng theo nhóm quyền

| Policy | Quyền | Lý do tách riêng |
| --- | --- | --- |
| `quillo-github-actions-deploy-policy` | ECR push (`quillo-api`), `autoscaling:StartInstanceRefresh` (scoped đúng ARN của ASG), `autoscaling:Describe*` (bắt buộc `Resource: "*"` — AWS không hỗ trợ resource-level permission cho action Describe*, cả 2 action đều read-only), S3 sync (`quillo-frontend-prod`) | Nhóm quyền chính, gắn trực tiếp cho vai trò deploy |
| `quillo-github-actions-lambda` | `lambda:UpdateFunctionCode`, `lambda:GetFunction` | Tách thành inline policy riêng — lần đầu setup, `put-role-policy` khi update policy chính đã **ghi đè mất** nhóm quyền Lambda vì dùng chung 1 policy. Bài học: mỗi nhóm quyền nên là 1 inline policy độc lập, tránh 1 lần update xóa mất nhóm khác. |

## Workflow: `ci.yml` — chạy mọi push/PR

- Lint + `tsc --noEmit` cho cả backend và frontend.
- Repo dùng npm workspaces với **1 package-lock.json** duy nhất ở root → `npm ci` chạy ở root, dùng `--workspace` flag thay vì `cd` vào từng thư mục con.
- `prisma generate` cần biến `DATABASE_URL` giả (chuỗi hợp lệ về format, không cần trỏ tới DB thật tồn tại) để pass bước validate schema.

## Workflow: `deploy.yml` — auto-deploy khi push `main`

```bash
check (lint + typecheck, gate)
│
├──▶ deploy-backend   ──┐
├──▶ deploy-lambda    ──┼──▶ smoke-test
└──▶ deploy-frontend  ──┘
```

| Job | Các bước chính |
| --- | --- |
| `deploy-backend` | `docker buildx build --platform linux/amd64` → push ECR (tag `:latest` + `:sha`) → `aws autoscaling start-instance-refresh` → poll trạng thái tới `Successful` (timeout 15 phút) |
| `deploy-lambda` | `build-lambda.sh` (esbuild bundle-all, cũng cần `DATABASE_URL` giả cho `prisma generate` bên trong) → `aws lambda update-function-code` |
| `deploy-frontend` | Set `VITE_API_BASE_URL` **trước** khi `vite build` (Vite bake biến môi trường vào lúc build, set sau vô nghĩa) → `aws s3 sync --delete` → gọi Cloudflare `purge_cache` API |
| `smoke-test` | `curl -f` health check cả backend và frontend, job đỏ nếu health check fail |

Hai lỗi thực tế gặp phải khi build pipeline:

- **`docker login` fail:** `ECR_REPO_URI` có kèm path (`.../quillo-api`), nhưng `docker login` chỉ nhận hostname registry → phải tách bằng `cut -d/ -f1` trước khi truyền vào lệnh login.
- **Cloudflare purge "thành công giả":** API Cloudflare trả **HTTP 200** ngay cả khi purge thật sự thất bại (`"success":false` nằm trong response body) → `curl -f` không đủ để phát hiện lỗi này, bắt buộc phải parse JSON response và kiểm tra field `success`.

## Kiểm chứng

```bash
gh run list --workflow=deploy.yml --limit 1
gh run view --log <run-id> | grep -i "smoke-test"

aws iam get-role --role-name quillo-github-actions-deploy-role \
  --query "Role.AssumeRolePolicyDocument.Statement[0].Condition"
```

> **Kết quả mong đợi:** `gh run list` cho thấy run gần nhất có status `completed` / conclusion `success`; log `smoke-test` chứa `"status":"ok"` cho cả backend và frontend; `get-role` trả về điều kiện trust policy giới hạn đúng `repo:Chubekho/Quillo:ref:refs/heads/main` (không phải `*`).
