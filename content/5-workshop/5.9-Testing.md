## End-to-End Business Flow Testing

Manual testing was performed on the actual production domain (`https://quillo.khuongle.site`), not a local/staging environment:

| Step | Action | Expected result |
| --- | --- | --- |
| 1 | Register a new organization (`POST /auth/register`) | Creates an `Organization` + `User` with role `OWNER`, auto-logs in |
| 2 | Create a Brand Persona (tone, keywords, target audience) | Persona saved to RDS, cached in Redis (30-minute TTL) |
| 3 | Create a Content Piece, select the persona, enter a brief, click Generate | Immediately returns `202 Accepted` with a `jobId` (no waiting for AI) |
| 4 | Frontend polls `GET /content/:id/jobs/:jobId` every 2.5s | Job transitions `QUEUED` → `PROCESSING` → `COMPLETED`, AI content is displayed |
| 5 | Export the content to PDF | The file correctly renders Vietnamese text (Noto Sans font), downloaded via a presigned URL |
| 6 | Check the Usage page | Tokens consumed increase to match the generation just performed |

## Quota Enforcement Testing

Repeated generation requests were sent until the test organization's `monthlyTokenQuota` was exceeded — confirming:

- The API returns `429` with a `QUOTA_EXCEEDED` message **before** the message is enqueued to SQS (avoiding wasted worker resources on a request that's guaranteed to be rejected).
- The worker also re-checks the quota before calling the AI provider — if exceeded, it marks the job `FAILED` and returns **without throwing an exception** (avoiding SQS treating it as a transient failure and auto-retrying, which would needlessly clog the queue).

## Daily Operations (Operational Runbook)

| Check | Command / where to look |
| --- | --- |
| Quick health check | `curl -f https://quillo-api.khuongle.site/api/v1/health` |
| ASG status | CloudWatch Alarms (section 5.7) — all 3 business alarms must be `OK` |
| Recent error logs | CloudWatch Logs Insights, log group `/quillo/api` or `/quillo/worker` |
| Stuck messages in the DLQ | `aws sqs get-queue-attributes --queue-url <dlq-url> --attribute-names ApproximateNumberOfMessages` — a non-zero value means a job failed more than 5 times and needs manual investigation |
| Unusual cost spikes | AWS Cost Explorer, filtered by the `Project=quillo` tag |

## Verification

```bash
curl -w "\n%{http_code}\n" -X POST https://quillo-api.khuongle.site/api/v1/content/<id>/generate \
  -H "Authorization: Bearer <token>"

aws sqs get-queue-attributes --queue-url <dlq-url> --attribute-names ApproximateNumberOfMessages
```

> **Expected result:** a valid generate request returns `202` with a `jobId`; a request exceeding the quota returns `429`; the DLQ's `ApproximateNumberOfMessages` is `0` (no stuck failed jobs).
