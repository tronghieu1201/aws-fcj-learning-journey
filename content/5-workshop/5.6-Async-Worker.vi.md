## Vì sao cần xử lý bất đồng bộ

Gọi AI để sinh nội dung mất vài giây đến vài chục giây — nếu để EC2 API chờ đồng bộ (đứng chờ Gemini trả lời rồi mới response cho client) sẽ chiếm dụng connection lâu, dễ timeout và không scale được khi nhiều user cùng generate. Giải pháp: API nhận request → đẩy job vào hàng đợi → trả về `202 Accepted` + `jobId` ngay lập tức → một tiến trình riêng (Lambda Worker) xử lý AI ở nền → client tự poll trạng thái job.

## Amazon SQS

| Queue | Vai trò |
| --- | --- |
| `quillo-generation-queue-prod` (Main Queue) | Nhận AI Job từ EC2 API, Standard Queue (không cần FIFO vì thứ tự xử lý không quan trọng) |
| `quillo-generation-dlq-prod` (Dead Letter Queue) | Nhận message xử lý thất bại quá số lần retry, qua Redrive Policy |

| Thông số | Giá trị |
| --- | --- |
| Visibility Timeout | 300 giây (đủ thời gian Lambda xử lý 1 job trước khi message bị coi là "chưa xử lý" và phát lại) |
| Redrive Policy | `maxReceiveCount = 5` → sau 5 lần fail, message tự động chuyển sang DLQ |

## AWS Lambda Worker

Lambda Worker viết bằng Node.js/TypeScript, chạy **trong VPC** (VPC-attached, gắn ENI vào Private App Subnet) để có thể kết nối RDS/Redis qua network nội bộ.

| Bước xử lý (`processMessage`) | Mô tả |
| --- | --- |
| 1. Nhận message | **Event Source Mapping** tự động poll Main Queue, invoke Lambda khi có message — Lambda không tự chủ động gọi SQS |
| 2. Check quota | Gọi `checkQuota()` — nếu tổ chức đã vượt hạn mức, đánh dấu job `FAILED` và return (không throw, để tránh retry/DLQ vô ích) |
| 3. Load dữ liệu | Truy vấn `ContentPiece` + `BrandPersona` từ RDS Primary qua Prisma |
| 4. Build prompt | Ghép Brand Persona (tone, voice, keywords, avoidWords) + Brief người dùng thành prompt hoàn chỉnh |
| 5. Gọi AI Provider | `ai.service.ts` dispatch theo `AI_PROVIDER=gemini` → gọi Gemini 2.5 Flash/Flash-Lite |
| 6. Lưu kết quả | Transaction: tạo `ContentVersion` mới (đánh dấu `isActive=true`, các version cũ set `false`), cập nhật `GenerationJob` = `COMPLETED`, ghi `UsageLog` (tokens tiêu thụ) |

Bundle triển khai bằng esbuild (`build-lambda.sh`), gộp toàn bộ dependency vào 1 file `index.js` (bundle-all strategy) để tránh lỗi thiếu module khi chạy trên Lambda runtime.

## Tích hợp Gemini API

- Lambda lấy `GEMINI_API_KEY` từ Secrets Manager (không hardcode).
- Model dùng: **Gemini 2.5 Flash** cho generate mới, **Gemini 2.5 Flash-Lite** cho các thao tác hậu kỳ (rewrite/expand/shorten) — Flash-Lite rẻ và nhanh hơn, phù hợp cho chỉnh sửa nhẹ trên nội dung đã có sẵn.
- Token usage đọc từ `usageMetadata.promptTokenCount` / `candidatesTokenCount` trong response, ghi vào `UsageLog` để phục vụ tính năng theo dõi quota.
- Lambda gọi ra Internet (Gemini API) thông qua NAT Gateway (đã mô tả ở mục [5.3 Networking](../5.3-networking/)).

## Kiểm chứng

```bash
aws sqs get-queue-attributes --queue-url <main-queue-url> \
  --attribute-names ApproximateNumberOfMessages ApproximateNumberOfMessagesNotVisible

aws lambda get-function --function-name quillo-worker --region ap-southeast-1 \
  --query "Configuration.[State,LastUpdateStatus]"

aws logs tail /aws/lambda/quillo-worker --since 1h --region ap-southeast-1
```

> **Kết quả mong đợi:** `get-queue-attributes` cho thấy số message đang chờ giảm dần về 0 sau khi job được xử lý; Lambda trả về `State: "Active"`, `LastUpdateStatus: "Successful"`; log CloudWatch chứa dòng xác nhận job hoàn tất, ví dụ có từ khóa `"status":"COMPLETED"` và không có `ERROR`/`Task timed out`.
