## Amazon RDS PostgreSQL (Multi-AZ)

| Parameter | Value |
| --- | --- |
| Engine | PostgreSQL 16 |
| Instance class | `db.t3.micro` |
| Multi-AZ | Enabled (Primary + Standby, automatic sync) |
| Subnet Group | Spans 2 private subnets (Data tier), 2 AZs |
| Connectivity | Only accepts connections from `ec2-sg` and `lambda-sg` on port 5432 |
| Encryption | Storage encrypted with an AWS KMS Customer Managed Key |
| SSL | Enforced (`force_ssl=1`); the application connects with `ssl: { rejectUnauthorized: false }` |

The application (EC2 API and Lambda Worker) only connects to the **RDS Primary endpoint** — there is no direct connection to the Standby. When the Primary fails, AWS automatically redirects the DNS endpoint to the Standby (failover); the application doesn't need to know the Standby exists.

The schema is managed with Prisma ORM (11 tables), migrated via `npx prisma migrate deploy` — run remotely through SSM Session Manager port-forwarding (port 5432 is never exposed to the internet).

## ElastiCache Redis

| Parameter | Value |
| --- | --- |
| Engine | Redis 7.x |
| Node type | `cache.t3.micro` |
| Location | Private subnet (Data tier), Security Group `redis-sg` (accepts only from `ec2-sg`, port 6379) |
| Used for | Caching Brand Persona (30-minute TTL), per-organization rate limiting |

Redis is only accessed by the **EC2 (Express API)** — the Lambda Worker does not connect to Redis, since asynchronous AI jobs read the persona directly from RDS when processing.

## AWS Secrets Manager

All sensitive secrets (JWT secret, database credentials, Gemini API key) are bundled into a **single JSON secret** (`quillo/app-secrets-prod`), instead of being read directly from a `.env` file in production.

```bash
USE_SECRETS_MANAGER=true   # production
USE_SECRETS_MANAGER=false  # local dev, reads directly from .env
```

The secret-loading mechanism (`loadSecrets()`) runs **before** any other module (database, JWT, AI provider) is imported — avoiding a race condition where `process.env` is read before the secret has finished loading.

EC2 and Lambda use separate IAM Roles (`quillo-ec2-role`, `quillo-lambda-role`) to read secrets — no static access key is ever stored on the instance or in the code.

## Verification

```bash
aws rds describe-db-instances --db-instance-identifier quillo-prod-db --region ap-southeast-1 \
  --query "DBInstances[0].[MultiAZ,DBInstanceStatus,Endpoint.Address]"

aws elasticache describe-cache-clusters --cache-cluster-id quillo-redis-prod --region ap-southeast-1

aws secretsmanager get-secret-value --secret-id quillo/app-secrets-prod --region ap-southeast-1 \
  --query "SecretString" --output text | jq 'keys'
```

> **Expected result:** the RDS command returns `true` (Multi-AZ) and `"available"`; the ElastiCache command returns `"CacheClusterStatus": "available"`; the Secrets Manager command returns exactly the 3 fields `["DATABASE_URL", "GEMINI_API_KEY", "JWT_SECRET"]` (no real values shown).
