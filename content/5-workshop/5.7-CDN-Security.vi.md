## Vì sao không dùng CloudFront — chênh lệch so với kế hoạch ban đầu

Kiến trúc dự tính ban đầu (xem [Proposal](/vi/2-proposal/)) dùng **Amazon CloudFront** làm CDN. Trong lúc triển khai thật, tài khoản AWS cá nhân gặp lỗi `AccessDenied` khi tạo CloudFront Distribution — nguyên nhân là tài khoản chưa **verify AWS Support Plan**, một yêu cầu bắt buộc với tài khoản mới trước khi được dùng một số dịch vụ nhất định. Đây là giới hạn khách quan ngoài tầm kiểm soát trong thời gian triển khai (4 tuần), nên đã pivot sang phương án thay thế: **Amazon S3 Static Website Hosting + Cloudflare** (free plan) — vẫn đáp ứng đủ yêu cầu chức năng (CDN, SSL, cache) cho quy mô demo.

## Frontend: S3 Static Website Hosting + Cloudflare

| Thành phần | Cấu hình |
| --- | --- |
| S3 Bucket | `quillo-frontend-prod`, bật Static Website Hosting, chứa build React (`dist/`) |
| Cloudflare | Free plan, DNS proxy (orange cloud) bật cho subdomain frontend |
| SSL Mode | Flexible (Cloudflare↔Origin không mã hóa end-to-end, chấp nhận được cho scope demo) |
| Domain frontend | `quillo.khuongle.site` |
| Domain backend | `quillo-api.khuongle.site` (subdomain 1 cấp, không phải `api.quillo.khuongle.site`) |

Hai vấn đề kỹ thuật gặp phải và cách xử lý:ư

- **Host header mismatch (S3 ↔ Cloudflare):** S3 Static Website trả lỗi `NoSuchBucket` vì Cloudflare gửi request với Host header không khớp tên bucket. Xử lý bằng **Cloudflare Cloud Connector** (miễn phí trên mọi plan) — không cần đổi tên bucket theo domain.
- **Cloudflare Universal SSL free chỉ phủ wildcard 1 cấp:** domain dạng `api.quillo.khuongle.site` (2 cấp) bị lỗi `ERR_SSL_VERSION_OR_CIPHER_MISMATCH`. Xử lý bằng cách đổi sang subdomain 1 cấp: `quillo-api.khuongle.site`.

> Known quirk (không ảnh hưởng chức năng): S3 Static Website Hosting trả HTTP status 404 cho route SPA fallback dù nội dung `index.html` vẫn load đúng — vì S3 không rewrite status code như CloudFront.

## AWS WAF

| Thông số | Giá trị |
| --- | --- |
| WebACL | `quillo-waf`, scope **REGIONAL** (không phải CLOUDFRONT, vì gắn trực tiếp vào ALB) |
| Managed Rule Groups | `AWSManagedRulesCommonRuleSet`, `AWSManagedRulesKnownBadInputsRuleSet` |
| Custom Rule | Rate limit 2000 request/5 phút/IP |
| Associated với | Application Load Balancer (`quillo-alb`) |

WAF chặn request **trước khi** chạm tới EC2 — đã kiểm thử thực tế bằng payload SQL Injection/XSS gửi trực tiếp qua domain thật, xác nhận trả về `403 Forbidden`.

## CloudWatch & SNS Alerting

| Log Group | Nguồn |
| --- | --- |
| `/quillo/api` | Log từ EC2 (winston-cloudwatch transport, chỉ bật khi `NODE_ENV=production`) |
| `/quillo/worker` | Log từ Lambda Worker |
| `/aws/waf/logs` | Log request bị WAF chặn |

| Metric Filter / Alarm | Điều kiện |
| --- | --- |
| API Error Rate | Số lỗi 5xx trong log `/quillo/api` vượt ngưỡng |
| Worker Error Rate | Số job `FAILED` trong log `/quillo/worker` vượt ngưỡng |
| WAF Blocked Requests | Số request bị WAF chặn tăng bất thường |

Cả 3 Alarm gửi thông báo qua **SNS Topic** (`quillo-prod-alerts`) tới email — cảnh báo chủ động thay vì phải tự kiểm tra log thủ công.

![AI Content Generator Platform architecture](/aws-fcj-learning-journey/images/5-Workshop/5.7-CDN-Security/cloudwatch.png) 

> **Lưu ý:** Alarm `TargetTracking-quillo-api-asg-AlarmLow-...` ở trạng thái "In alarm" là **hành vi thiết kế bình thường**, không phải sự cố — đây là alarm nội bộ do Target Tracking Scaling Policy của Auto Scaling Group tự tạo, cố ý kích hoạt khi CPU trung bình xuống thấp để ra lệnh scale-in (giảm bớt instance khi tải thấp). 3 alarm nghiệp vụ thật (`quillo_api_error_rate_high`, `quillo_worker_error_rate_high`, `quillo_waf_blocked_requests_high`) đều ở trạng thái `OK`.

## Kiểm chứng

```bash
curl -I https://quillo.khuongle.site
curl "https:r//quillo-api.khuongle.site/api/v1/health?id=1' OR '1'='1"

aws wafv2 get-web-acl --name quillo-waf --scope REGIONAL --region ap-southeast-1 \
  --query "WebACL.Rules[].Name"

aws cloudwatch describe-alarms --alarm-name-prefix quillo --region ap-southeast-1 \
  --query "MetricAlarms[].[AlarmName,StateValue]"
```

> **Kết quả mong đợi:** `curl` domain frontend trả `HTTP/2 200`; `curl` chứa payload SQL Injection trả về `403 Forbidden` (WAF chặn); `get-web-acl` liệt kê đủ 2 Managed Rule Group + rule rate-limit; `describe-alarms` trả về `StateValue: "OK"` cho cả 3 alarm (không phải `ALARM`).
