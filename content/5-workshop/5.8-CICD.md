## AWS Authentication via OIDC — No Long-Lived Access Keys

GitHub Actions authenticates with AWS via **OpenID Connect (OIDC)**, entirely removing the need to store `AWS_ACCESS_KEY_ID`/`AWS_SECRET_ACCESS_KEY` as GitHub Secrets — those are static keys that persist indefinitely until manually revoked, a far greater risk than the temporary credentials OIDC issues for each workflow run.

| Component | Configuration |
| --- | --- |
| OIDC Identity Provider | `token.actions.githubusercontent.com` |
| IAM Role | `quillo-github-actions-deploy-role` |
| Trust Policy | Only accepts tokens from the `Chubekho/Quillo` repo, branch `main` — workflows running from another branch or a fork cannot obtain credentials |

> Setup note: the AWS CLI **requires** the `--thumbprint-list` parameter when creating an OIDC provider, even though this value hasn't actually been verified by GitHub Actions since 2023 — omitting it causes the `create-open-id-connect-provider` command to fail.

## IAM Policy — Split by Permission Group

| Policy | Permissions | Reason for separation |
| --- | --- | --- |
| `quillo-github-actions-deploy-policy` | ECR push (`quillo-api`), `autoscaling:StartInstanceRefresh` (scoped to the exact ASG ARN), `autoscaling:Describe*` (requires `Resource: "*"` — AWS doesn't support resource-level permissions for Describe* actions, both are read-only), S3 sync (`quillo-frontend-prod`) | The main permission group, attached directly to the deploy role |
| `quillo-github-actions-lambda` | `lambda:UpdateFunctionCode`, `lambda:GetFunction` | Split into a separate inline policy — during initial setup, a `put-role-policy` update to the main policy **overwrote** the Lambda permission group because both shared one policy. Lesson learned: each permission group should be its own inline policy, so one update doesn't wipe out another. |

## Workflow: `ci.yml` — Runs on Every Push/PR

- Lint + `tsc --noEmit` for both backend and frontend.
- The repo uses npm workspaces with a **single package-lock.json** at the root → `npm ci` runs at the root, using the `--workspace` flag instead of `cd`-ing into each subdirectory.
- `prisma generate` needs a dummy `DATABASE_URL` (a format-valid string, not pointing to a real existing database) to pass schema validation.

## Workflow: `deploy.yml` — Auto-Deploy on Push to `main`

```bash
check (lint + typecheck, gate)
│
├──▶ deploy-backend   ──┐
├──▶ deploy-lambda    ──┼──▶ smoke-test
└──▶ deploy-frontend  ──┘
```

| Job | Main steps |
| --- | --- |
| `deploy-backend` | `docker buildx build --platform linux/amd64` → push to ECR (tags `:latest` + `:sha`) → `aws autoscaling start-instance-refresh` → poll status until `Successful` (15-minute timeout) |
| `deploy-lambda` | `build-lambda.sh` (esbuild bundle-all strategy, also needs a dummy `DATABASE_URL` for the internal `prisma generate`) → `aws lambda update-function-code` |
| `deploy-frontend` | Sets `VITE_API_BASE_URL` **before** running `vite build` (Vite bakes environment variables in at build time; setting it after is meaningless) → `aws s3 sync --delete` → calls the Cloudflare `purge_cache` API |
| `smoke-test` | `curl -f` health checks on both backend and frontend; the job fails if either health check fails |

Two real issues encountered while building the pipeline:

- **`docker login` failure:** `ECR_REPO_URI` includes a path (`.../quillo-api`), but `docker login` only accepts the registry hostname → it must be split with `cut -d/ -f1` before being passed to the login command.
- **False-positive Cloudflare purge:** Cloudflare's API returns **HTTP 200** even when the purge actually fails (`"success":false` in the response body) → `curl -f` alone can't catch this; the JSON response body must be parsed and the `success` field checked.

## Verification

```bash
gh run list --workflow=deploy.yml --limit 1
gh run view --log <run-id> | grep -i "smoke-test"

aws iam get-role --role-name quillo-github-actions-deploy-role \
  --query "Role.AssumeRolePolicyDocument.Statement[0].Condition"
```

> **Expected result:** `gh run list` shows the latest run with status `completed` / conclusion `success`; the `smoke-test` log contains `"status":"ok"` for both backend and frontend; `get-role` returns a trust policy condition scoped to exactly `repo:Chubekho/Quillo:ref:refs/heads/main` (not `*`).
