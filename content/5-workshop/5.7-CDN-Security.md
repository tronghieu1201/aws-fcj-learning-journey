## Why Not CloudFront — A Deviation from the Original Plan

The original architecture (see the [Proposal](/2-proposal/)) planned to use **Amazon CloudFront** as the CDN. During actual deployment, the personal AWS account hit an `AccessDenied` error when creating a CloudFront Distribution — the root cause was that the account had not **verified an AWS Support Plan**, a prerequisite for certain services on new accounts. This was an objective constraint outside our control within the 4-week implementation window, so the team pivoted to an alternative: **Amazon S3 Static Website Hosting + Cloudflare** (free plan) — which still meets the functional requirements (CDN, SSL, caching) for a demo-scale deployment.

## Frontend: S3 Static Website Hosting + Cloudflare

| Component | Configuration |
| --- | --- |
| S3 Bucket | `quillo-frontend-prod`, Static Website Hosting enabled, holds the React build (`dist/`) |
| Cloudflare | Free plan, DNS proxy (orange cloud) enabled for the frontend subdomain |
| SSL Mode | Flexible (Cloudflare↔Origin is not end-to-end encrypted, acceptable for demo scope) |
| Frontend domain | `quillo.khuongle.site` |
| Backend domain | `quillo-api.khuongle.site` (single-level subdomain, not `api.quillo.khuongle.site`) |

Two technical issues encountered and how they were resolved:

- **Host header mismatch (S3 ↔ Cloudflare):** S3 Static Website returned `NoSuchBucket` because Cloudflare sent requests with a Host header that didn't match the bucket name. Resolved using **Cloudflare Cloud Connector** (free on every plan) — no need to rename the bucket to match the domain.
- **Cloudflare's free Universal SSL only covers a single-level wildcard:** a two-level domain like `api.quillo.khuongle.site` failed with `ERR_SSL_VERSION_OR_CIPHER_MISMATCH`. Resolved by switching to a single-level subdomain: `quillo-api.khuongle.site`.

> Known quirk (doesn't affect functionality): S3 Static Website Hosting returns an HTTP 404 status for the SPA fallback route even though the `index.html` content still loads correctly — because S3 doesn't rewrite status codes the way CloudFront does.

## AWS WAF

| Parameter | Value |
| --- | --- |
| WebACL | `quillo-waf`, scope **REGIONAL** (not CLOUDFRONT, since it's attached directly to the ALB) |
| Managed Rule Groups | `AWSManagedRulesCommonRuleSet`, `AWSManagedRulesKnownBadInputsRuleSet` |
| Custom Rule | Rate limit of 2,000 requests/5 minutes/IP |
| Associated with | Application Load Balancer (`quillo-alb`) |

WAF blocks requests **before** they reach EC2 — verified in practice by sending SQL Injection/XSS payloads directly through the real domain and confirming a `403 Forbidden` response.

## CloudWatch & SNS Alerting

| Log Group | Source |
| --- | --- |
| `/quillo/api` | Logs from EC2 (winston-cloudwatch transport, enabled only when `NODE_ENV=production`) |
| `/quillo/worker` | Logs from the Lambda Worker |
| `/aws/waf/logs` | Logs of requests blocked by WAF |

| Metric Filter / Alarm | Condition |
| --- | --- |
| API Error Rate | 5xx errors in `/quillo/api` logs exceed a threshold |
| Worker Error Rate | `FAILED` jobs in `/quillo/worker` logs exceed a threshold |
| WAF Blocked Requests | An abnormal spike in requests blocked by WAF |

All 3 alarms send notifications via an **SNS Topic** (`quillo-prod-alerts`) to email — proactive alerting instead of manually checking logs.

![AI Content Generator Platform architecture](/aws-fcj-learning-journey/images/5-Workshop/5.7-CDN-Security/cloudwatch.png) 

> **Note:** The `TargetTracking-quillo-api-asg-AlarmLow-...` alarm showing "In alarm" is **expected behavior, not an incident** — it's an internal alarm auto-created by the Auto Scaling Group's Target Tracking Scaling Policy, intentionally triggered when average CPU drops low to signal a scale-in (reducing instance count under low load). The 3 real business alarms (`quillo_api_error_rate_high`, `quillo_worker_error_rate_high`, `quillo_waf_blocked_requests_high`) are all in the `OK` state.

## Verification

```bash
curl -I https://quillo.khuongle.site
curl "https://quillo-api.khuongle.site/api/v1/health?id=1' OR '1'='1"

aws wafv2 get-web-acl --name quillo-waf --scope REGIONAL --region ap-southeast-1 \
  --query "WebACL.Rules[].Name"

aws cloudwatch describe-alarms --alarm-name-prefix quillo --region ap-southeast-1 \
  --query "MetricAlarms[].[AlarmName,StateValue]"
```

> **Expected result:** `curl` on the frontend domain returns `HTTP/2 200`; the `curl` request containing the SQL Injection payload returns `403 Forbidden` (blocked by WAF); `get-web-acl` lists both managed rule groups plus the rate-limit rule; `describe-alarms` returns `StateValue: "OK"` for all 3 alarms (not `ALARM`).
